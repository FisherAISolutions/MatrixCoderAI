# CodePilot — Patch Bug Fix + GitHub Import (Handoff Report)

## 1. Root-Cause Analysis

The AI agent's `edit:` SEARCH/REPLACE pipeline appeared to succeed (toast +
chat message both said "Patched file successfully") but the file viewer
kept showing stale content. Investigation revealed **three distinct root
causes** stacked on top of each other:

### Root cause #1 — Path normalization mismatch (extractors)
`extractEdits` in `src/lib/repo/extractors.ts` only `.trim()`-ed the raw
path captured from the `\`\`\`edit:<path>` fence. The AI variably emits any
of: `./src/foo.ts`, `src\foo.ts`, `/src/foo.ts`, `"src/foo.ts"`. Each of
those is a different string, and the downstream `byPath.get(e.path)`
lookup in `ChatComposer.tsx` did an exact string compare — so the patch
*looked* like a successful no-op when the model added a leading `./` or
mismatched separators.

### Root cause #2 — Lookup did not tolerate `src/` prefix drift
Even after normalization, the AI sometimes emits `app/layout.tsx` when
the tree stores `src/app/layout.tsx` (Next.js apps often live under
`src/`, but the model isn't always told). The original `byPath.get(...)`
did exact-only lookup with no fallback.

### Root cause #3 — Stale `viewedFile` in `FileTreeSidebar`
`FileTreeSidebar.tsx` keeps its own `viewedFile` local state set at
click-time, then renders **its own** `<FileViewer file={viewedFile}>`
overlay. After AI patches, `ChatWorkspacePage.activeFile` and `fileTree`
both update correctly — but `viewedFile` was never refreshed, so the
sidebar's viewer continued to show the pre-patch content (and it overlays
the workspace-level viewer in some layouts, so the user perceived "no
update").

### Additional sub-issue — `updateFile` destructuring drift
`setFileTree(... updateInTree ...)` used `{ ...n, ...file }` which lets
`file.id` overwrite the tree node's stable id. When a patch was matched
by path-only (because the AI didn't know the row id), this clobbered
React keys and could orphan the node in subsequent updates. Same problem
applied to `setActiveFile`.

## 5. Anything Still Potentially Broken / Out of Scope

- **Private repos**: deferred per user's choice (PAT support is a 5-line
  change in `githubImport.ts` — already noted with a comment).
- **GitHub rate limits**: unauthenticated calls are limited to ~60/hour
  per IP. Large repos can exhaust this on the metadata + tree calls (raw
  blob fetches use a separate CDN with no API limit). Error is surfaced
  inline in the modal.
- **GitHub tree truncation**: GitHub truncates the recursive tree at
  ~100k entries / 7 MB. For repos this large the import will log a
  warning and proceed with the partial tree. (Out of scope: paginating
  via per-subdirectory `git/trees` walks.)
- **Patch debug logs**: kept in (per user choice) prefixed with
  `[extractor]`, `[patcher]`, `[updateFile]`, `[FileTreeSidebar]`,
  `[FileViewer]`. Filter the console by any of those prefixes for
  targeted debugging.

  page): kept as-is per surgical-fix constraint. The sync effect added

## 8. Iteration 2 — Full Scaffold Generation Fix

The Coding Agent emitted only component/page files when asked to "build
an app", because the extractor silently dropped root-level files. Two
fixes shipped together:

### Extractor (`src/lib/repo/extractors.ts`)
- `CREATE_WITH_PATH_REGEX` now accepts any file path (not just paths
  ending in a hardcoded extension list) and any comment style
  (`// path:`, `# path:`, `<!-- path: ... -->`, `/* path: ... */`).
- Dropped the `cleanPath.startsWith('src/')` gate that silently rejected
  `package.json`, `README.md`, `tailwind.config.ts`, `next.config.mjs`,
  `postcss.config.js`, `tsconfig.json`, `app/globals.css`, etc.
- New `sanitizeCreatePath()` accepts root-level files but rejects path
  traversal (`../`), URLs, absolute Windows paths, and known system
  folders (`/etc`, `/usr`, …, `windows`, …).
- `INLINE_PATH_REGEX` (fallback for bare fences) now recognises the
  most common project subtrees AND a curated list of root config files.
- `extractMentionedPaths()` likewise expanded so the file-chip rail
  underneath assistant messages picks up root-level mentions.
- Pass 2 (bare-fence inference) refuses to substitute a fallback path
  when the fence had an explicit (but rejected) `// path:` annotation —
  closes a sneaky avenue where `// path: /etc/passwd` could have been
  rewritten to `src/generated0.ts`.

### Coding Agent system prompt (`ChatComposer.tsx`)
The `coding` agent prompt now contains a dedicated **NEW-APP SCAFFOLDING**
section that:
- Lists the exact minimum files for Next.js 15 App Router (`package.json`,
  `next.config.mjs`/`.ts`, `tsconfig.json`, `tailwind.config.ts`/`.js`,
  `postcss.config.js`, `app/layout.tsx`, `app/page.tsx`,
  `app/globals.css`, `components/*`, `lib/*`, `types/*`, `README.md`).
- Lists the exact minimum files for Vite + React (`package.json`,
  `index.html`, `src/main.tsx`, `src/App.tsx`, `src/index.css`,
  `vite.config.ts`, `tsconfig.json`, `tailwind.config.ts`,
  `postcss.config.js`).
- Forbids mixing Next + Vite in one project.
- Forbids leading `/` or `./` on emitted paths.
- Says root config files are FIRST-CLASS — emit them with the same
  ```` ```<lang>\n// path: ... ```` pattern as components.
- Tells the agent to pin recent stable major versions in `package.json`
  (`next ^15`, `react ^19`, `tailwindcss ^3.4`).
- Reiterates that when the repo context already lists a file, it must
  be patched via `edit:` (SEARCH/REPLACE), not regenerated — preserves
  the patch pipeline behaviour.
